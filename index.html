Has AI Been Looking in the Wrong Direction for Its Architecture?

The 5G Core's Service-Based Architecture already solved the orchestration, memory, and governance problems that Agentic AI is wrestling with today. Here is the complete mapping.
VM
Vinit Metange
AI PM  ·  Telecom-AI Architecture Research

The conversation about AI in telecom has been almost entirely one-directional. AI will optimize the RAN. AI will predict network faults. AI will automate the NOC. These are legitimate and valuable applications, and significant work is being done on all of them.

But while spending time studying both domains simultaneously, a different question surfaced. Not "what can AI do for telecom?" but its inverse: what can telecom teach AI?

Specifically: what can the 3GPP Service-Based Architecture of the 5G Core teach the people building the next generation of enterprise multi-agent AI systems?

The answer turned out to be more substantive than expected. Not a loose metaphor. An actual architectural mapping, function by function, design principle by design principle.

The 5G Core is, structurally, a multi-agent AI system. It just doesn't call itself one.

The Problem with How AI Learns from Telecom

The typical AI-telecom crossover conversation goes like this: telecom engineers explain their network problems, AI researchers propose solutions, and the result is AIOps, AI-native OSS, or autonomous network management. The knowledge flows in one direction.

This framing is limiting. It treats telecom as a problem domain and AI as the solution domain. It misses the possibility that telecom's architecture, developed over decades under the most demanding reliability requirements in any commercial software system, might contain structural lessons that AI builders haven't yet internalized.

Agentic AI systems in 2026 are dealing with a specific set of hard problems:

The hard problems of Agentic AI: How do you coordinate multiple specialized agents without a monolith? How do you manage state across agent boundaries? How do you enforce policy without baking it into every agent? How do you discover and route to the right capability at runtime? How do you do all of this at scale, with fault tolerance, and under continuous observability?

These are not new problems. They are the problems that distributed systems engineers have been solving for decades. And telecom's 3GPP standards body formalized exceptionally thorough answers to each of them in the 5G Core specification, codified in 3GPP TS 23.501 and the broader Release 15 through Release 19 architecture documents.

The Architectural Insight

The 5G Core is built on a Service-Based Architecture (SBA). Each Network Function (NF) is a microservice: independent, stateless by design, horizontally scalable, communicating via RESTful APIs over the Service-Based Interface (SBI). The NFs don't know about each other's internals. They register with a central registry, expose well-defined APIs, and communicate through those APIs with explicit policy enforcement at every boundary.

When you map this against a modern multi-agent AI system, the structural correspondence is not approximate. It is precise.

5G NF Full Name Agentic AI Equivalent Function
AMF Access & Mobility Mgmt Function → Supervisor Agent Workflow coordination, lifecycle management
SMF Session Management Function → Planning Agent Transforms intent into an executable plan
UPF User Plane Function → Execution Agent Runs tasks without reasoning about them
UDM + UDR Unified Data Mgmt + Repository → Long-Term Memory Persistent identity, subscription, history
UDSF Unstructured Data Storage Function → Shared Context Memory Externalized runtime agent state
NRF Network Repository Function → Service Registry Dynamic NF / agent discovery
NEF Network Exposure Function → MCP Gateway Secure external API and tool exposure
AUSF Authentication Server Function → Identity Layer Authentication, trust establishment
PCF Policy Control Function → Guardrails Engine Policy, QoS, access control
NWDAF Network Data Analytics Function → AI Observability Telemetry, anomaly detection, forecasting
SCP Service Communication Proxy → Service Mesh Routing, load balancing, resilience
NSSF Network Slice Selection Function → Agent Router Intent-based routing to capability
CHF Charging Function → Token / Cost Metering Usage accounting per session

The terminology is completely different. The architectural patterns are the same.

Network Function by Network Function: Why Each Mapping Holds

Calling this a mapping requires justification. Here is the architectural reasoning for each one.

AMF
Access & Mobility Management Function → Supervisor Agent
The AMF does not carry user traffic. Its job is orchestration: managing the registration process, coordinating other NFs, maintaining context across interactions, and handling lifecycle events. It invokes SMF, AUSF, UDM, and PCF in sequence to complete a registration. This is the definition of a Supervisor Agent: it coordinates specialist functions without doing the specialist work itself.
SMF
Session Management Function → Planning Agent
The SMF takes a session request (intent) and transforms it into an execution plan: select the UPF, determine the QoS policy, establish the tunnel, assign the IP. It doesn't execute the plan, the UPF does. The SMF is a pure planning layer. This is precisely what a Planning Agent does in a multi-agent AI architecture: transform high-level intent into a sequence of concrete, executable steps.
UPF
User Plane Function → Execution Agent / Tool Runner
The UPF forwards packets. It inspects, routes, reports. It does not reason about what it is doing or why. It executes instructions with high throughput and low latency. This is the Tool Execution Agent in Agentic AI: a specialized function that takes a well-defined action and returns a result, without maintaining its own planning context.
UDM + UDR
Unified Data Mgmt + Repository → Long-Term Memory
UDM/UDR stores persistent user information: subscriber identity, authentication credentials, subscription profiles, policy associations, and service preferences. This data persists across sessions and is consulted at multiple points in the workflow. In AI terms, this is long-term memory: the persistent knowledge store that gives an agent context about a user, their history, and their preferences, across sessions, across time.
UDSF
Unstructured Data Storage Function → Shared Context Memory
This is one of the most architecturally precise mappings. The UDSF externalizes transient runtime state, NF instance data, session context, so that any stateless NF instance can retrieve current context without maintaining it internally. In cloud-native AI platforms, this is the role of shared context memory (Redis, Blackboard architecture): a fast, externalized state store that allows stateless agents to maintain continuity across invocations.
NRF
Network Repository Function → Service Registry / MCP Registry
NFs register with the NRF on startup and deregister on shutdown. When an NF needs to invoke another, it queries the NRF for the current set of available instances and their endpoints. This is dynamic service discovery, the same pattern implemented by Kubernetes service discovery, HashiCorp Consul, and, in the agentic AI space, by MCP registries and agent capability registries.
AUSF
Authentication Server Function → Identity Layer
Every AI system that operates on behalf of real users must solve identity. The AUSF authenticates subscribers during registration, establishes trust, and generates security keys. The architectural requirement, verify identity before granting access to any capability, is identical in enterprise AI systems. The AUSF is the telecom instance of what AI platforms call the identity provider.
PCF
Policy Control Function → Guardrails Engine
The PCF defines and enforces policies that govern what each NF is permitted to do: QoS constraints, access control rules, charging policies, usage limits. It acts as a central policy authority that is consulted across multiple flows. In agentic AI, this is the Guardrails Engine: the layer that enforces what agents are permitted to do, what tools they can invoke, what data they can access, and within what constraints they must operate.
NWDAF
Network Data Analytics Function → AI Observability Platform
The NWDAF is the most architecturally forward-looking NF in the 5G specification. It continuously collects telemetry from across the network, detects anomalies, forecasts load, and provides recommendations to other NFs. In 3GPP Release 18, its role expands to include closed-loop automation. In AI terms: this is exactly what a modern AI observability platform does, collect traces, detect drift, forecast issues, and trigger remediation.

Why This Is Not Just an Analogy

The mappings above are not superficial naming similarities. They hold because both architectures are solving the same fundamental distributed systems problem: how do you coordinate specialized, independent functions at scale, with fault tolerance, under continuous policy enforcement, with shared state and independent scalability?

Both architectures converged on the same set of software engineering principles:

Specialization over monoliths
API-first communication
Stateless compute
Externalized shared state
Dynamic service discovery
Policy-driven governance
Event-driven coordination
Continuous observability
Independent scalability
Autonomous decision support
The key insight: These are not coincidental overlaps. Microservices architecture, API-first communication, external state management, and policy enforcement are universal patterns for large-scale distributed systems. 5G adopted them comprehensively in the early 2010s, formalized them in 3GPP TS 23.501, and proved them at global scale. Agentic AI is now independently re-deriving the same architectural conclusions.

What AI Builders Can Steal from Telecom

Telecom's decades of carrier-grade engineering produced specific lessons that AI system architects haven't yet fully internalized:

1. The separation of plane and state

5G strictly separates the control plane (which reasons and coordinates) from the user plane (which executes). State is externalized rather than held by compute functions. This separation is what enables horizontal scaling. Many current agentic AI systems conflate these concerns, agents hold their own state, making them hard to scale and resilient to failure. The UDSF pattern (externalized runtime state) directly addresses this.

2. Policy as a first-class architectural concern

In the 5G Core, the PCF is not an afterthought. It is a dedicated NF that is consulted in multiple flows. Policy is not baked into individual NFs, it is centralized and enforced at boundaries. AI builders are still figuring out where guardrails live. The telecom answer: they live in a dedicated, centralized Policy Control Function that all flows consult.

3. Five-nines is a design constraint, not a product goal

Telecom systems are engineered for 99.999% availability as a specification requirement, not an aspiration. This forces specific patterns: stateless compute, redundant instances, circuit breakers, graceful degradation, automated failover. Enterprise AI systems that are intended to operate autonomously need to adopt the same posture. An agentic system that fails silently or non-gracefully cannot be trusted with autonomous decision-making.

4. Closed-loop automation as the end state

3GPP Release 18's NWDAF extensions move toward closed-loop automation: detect an anomaly, determine a remediation, apply it, and validate the outcome without human intervention. This is exactly the end state that agentic AI systems are being designed toward. Telecom has a head start on the governance, rollback, and audit trail requirements that make closed-loop automation trustworthy.

This is not hypothetical. This is what carrier-grade enterprise agentic AI will need to look like. The question is whether AI builders will discover these patterns by re-deriving them from first principles over the next decade, or whether they will recognize that the 5G Core spec contains a proven, battle-tested implementation guide right now.

The Question Worth Sitting With

For years, telecom has looked toward AI for its transformation. The framing has been: "AI will help telecom evolve." That's true. But the reverse is equally true, and far less discussed.

The Service-Based Architecture of the 5G Core proves that every major architectural challenge in enterprise multi-agent AI, orchestration, memory, governance, observability, resilience, has already been solved, at global scale, under the most demanding reliability requirements in commercial software.

The next generation of enterprise Agentic AI may owe more to 5G architecture than the field currently realizes.

The people who understand both domains, who can read 3GPP TS 23.501 and a LangGraph system design in the same sitting and recognize what they share, are going to build substantially better AI systems than those who only know one side.

If you are building agentic AI for enterprise and haven't read the 5G Core architecture specification: it is worth your time. Not because telecom is the answer. But because telecom already found it.
